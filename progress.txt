## 2026-02-24 — Refkey Helpers (Phase 0, Task 0.4)

### What was done
- Created `src/utils/refkeys.ts`: 14 refkey helper functions covering all artifact types
  needed by the emitter — types, serializers, deserializers, polymorphic types, known values,
  operation options, client artifacts (context, factory, classical class), XML serializers/
  deserializers, and utility helpers (serialization, paging, polling).
- Created `src/utils/index.ts`: Barrel file exporting all refkey helpers.
- Updated `src/index.ts`: Re-exports the utils module.
- Created `test/utils/refkeys.test.tsx`: 8 tests covering:
  - Identity/memoization (same entity → same refkey)
  - Uniqueness (different discriminators → different refkeys)
  - Entity isolation (different entities → different refkeys)
  - Helper name isolation (different names → different refkeys)
  - Cross-category isolation (same name, different category → different refkeys)
  - Base refkey vs discriminated refkeys
  - XML vs JSON serializer refkey distinction
  - Client-related refkey distinction (context, factory, classical)
- All 19 tests pass (8 refkey + 5 infrastructure + 6 context).

### Architecture decisions
- Entity-based helpers (typeRefkey, serializerRefkey, etc.) accept `unknown` entities — they
  wrap `refkey(entity, discriminator)` from `@alloy-js/core` with descriptive discriminator strings.
- `typeRefkey(entity)` uses no discriminator — it returns `refkey(entity)` directly, making it
  the "identity" refkey for any entity.
- String-based helpers (serializationHelperRefkey, pagingHelperRefkey, pollingHelperRefkey)
  accept a name string and use a category prefix as the first argument to prevent cross-category
  collisions (e.g., `refkey("serializationHelper", name)`).
- Discriminator strings follow the legacy emitter's naming convention for consistency.

## 2026-02-24 — Type Expression Component (Phase 1, Task 1.1)

### What was done
- Created `src/components/type-expression.tsx`: `getTypeExpression()` function that maps
  TCGC `SdkType` to TypeScript type expressions (Alloy `Children`).
- Updated `src/components/index.ts`: Exports `getTypeExpression` from barrel file.
- Created `test/components/type-expression.test.tsx`: 13 tests covering:
  - Primitive scalars (string, int32, float32, boolean, bytes, utcDateTime)
  - All numeric types (int8, int16, int64, uint8-64, float64) → number
  - plainDate → Date
  - offsetDateTime → string
  - duration → string (default encoding)
  - Array types → (T)[] syntax
  - Dictionary types → Record<string, T>
  - Nullable types → T | null
  - Nested composite types (int32[][] → ((number)[])[] )
  - Model reference via refkey (renders as model name)
  - Enum reference via refkey (renders as enum name)
  - String constant literals → quoted strings
  - Numeric constant literals → unquoted numbers
- All 43 tests pass (13 new + 30 existing).

### Architecture decisions
- **Function, not JSX component**: `getTypeExpression` returns `Children` (string, refkey,
  or code template) rather than being a JSX component. This is because it's a type
  mapping utility consumed by other components, not a standalone renderable unit.
- **Legacy emitter parity**: All type mappings match the legacy emitter's behavior exactly.
  Notably, int64/uint64 → "number" (not bigint) for output parity with existing SDKs.
- **No context dependency**: The function is pure — it only needs the SdkType, not any
  Alloy context. This makes it easy to test and compose.
- **Refkeys for named types**: Model, enum, and union types return `typeRefkey(entity)`
  which enables Alloy's automatic cross-file import resolution.

## 2026-02-24 — Model Interface Component (Phase 1, Task 1.2)

### What was done
- Created `src/components/model-interface.tsx`: `ModelInterface` JSX component that generates
  `<InterfaceDeclaration>` from TCGC `SdkModelType`.
- Updated `src/components/index.ts`: Added `ModelInterface` export.
- Created `test/components/model-interface.test.tsx`: 9 tests covering:
  - Basic model with string/number/boolean properties
  - Optional properties rendered with `?` modifier
  - Readonly properties (visibility=[Read]) rendered with `readonly`
  - Base model inheritance via `extends` clause using refkey
  - Discriminator property on subtype renders literal `discriminatorValue` type
  - additionalProperties render as index signature `[key: string]: T`
  - JSDoc documentation on interface from model.doc
  - JSDoc documentation on members from property.doc
  - Cross-model reference types resolved via refkey
- All 52 tests pass (9 new + 43 existing).

### Architecture decisions
- **JSX component, not function**: Unlike `getTypeExpression()` (a function), `ModelInterface`
  is a proper JSX component because it renders structural output (`<InterfaceDeclaration>`)
  that composes child components. It's a middle-tier component in the component hierarchy.
- **Helper functions**: Internal logic split into small, documented helper functions:
  - `getExtendsClause()` — builds `extends` clause from `model.baseModel` via refkey
  - `getModelDoc()` — extracts doc string from `model.doc` or `model.summary`
  - `getExpandedProperties()` — collects properties with flatten expansion
  - `getPropertyTypeExpression()` — resolves property type, handling discriminator literals
  - `isReadOnly()` — checks visibility for read-only detection (reimplements TCGC's internal utility)
- **Flattened properties**: When `property.flatten === true` and the property type is a model,
  the nested model's properties are expanded inline. Children inherit the wrapper's optionality.
- **Discriminator handling**: On subtypes (models with `discriminatorValue`), the discriminator
  property type becomes the string literal `"value"` instead of the general type.
- **additionalProperties**: Rendered as a separate `<InterfaceMember indexer=...>` after the
  `<For>` block for regular properties, with a newline separator.

### Key learnings (see knowledge.md)
- Alloy JSDoc always renders multi-line format (`/**\n * text\n */`), not single-line.
- TCGC subtypes only contain their own properties, not inherited ones from baseModel.
- Index signature members outside `<For>` don't get automatic trailing semicolons.
