## 2026-02-26: RC30 Split — Property Flatten Serialization (Copilot)

### Analysis Performed
Deep analysis of RC30 (Property flatten serialization approach) revealed three distinct sub-problems that should be addressed separately:

1. **Wire Format (RC30a, P1)**: The serializer puts flattened properties at the top level (`{ result, bar, baz }`) instead of wrapping them under the original wire name (`{ result, properties: { bar, baz } }`). Similarly, the deserializer reads from `item["bar"]` instead of `item["properties"]["bar"]`. This is a correctness bug affecting the HTTP request/response body structure.

2. **Name Collision (RC30b, P1)**: When a parent model has properties with the same name as flattened child properties (e.g., both have `bar`), the current emitter produces duplicate property names in interfaces and serializers (invalid TypeScript). Legacy resolves this by prefixing collisions with the flatten property path (e.g., `barPropertiesBar`).

3. **Helper Functions (RC30c, P2)**: Legacy generates separate exported helper functions (`_testPropertiesSerializer`, `_testPropertiesDeserializer`) for each flatten group. The new emitter can inline the nested object for correct wire format, but generating separate helpers achieves full output parity.

### Key Findings from Legacy Emitter Analysis
- **Serializer pattern**: `properties: _testPropertiesSerializer(item)` for required, with `areAllPropsUndefined(item, [...])` guard for optional
- **Deserializer pattern**: Uses spread `..._testPropertiesDeserializer(item["properties"])` to merge deserialized flatten properties
- **areAllPropsUndefined** helper already exists in new emitter's static helpers
- **Name collision resolution**: First flatten property wins naming, subsequent collisions get `{propName}{FlattenPropName}{PropName}` prefix pattern

### PRD Changes
- RC30 split into: RC30a (wire format, P1), RC30b (name collision, P1), RC30c (helper functions, P2)

## 2026-02-26 — SA-C26: Default Values for Query Parameters and Headers (done)

### What was done
- Added `??` default value fallbacks for optional query parameters, header parameters, and body parameters that have `clientDefaultValue` set via TCGC's `@clientDefaultValue` decorator
- Legacy emitter applies `options?.paramName ?? defaultValue` pattern; new emitter now matches this behavior
- Implemented `applyClientDefault()`, `formatDefaultValue()`, and `isDefaultValueTypeMatch()` helper functions in `send-operation.tsx`

### Key decisions
- Only apply `??` defaults when ALL of: `param.optional === true`, `clientDefaultValue` is defined, AND type matches (string→string, number→numeric, boolean→boolean)
- Required params with `@clientDefaultValue` go to options bag but do NOT get `??` (matches scenario test expectations)
- API version parameters excluded via `isApiVersionParam` check (they're managed by client infrastructure, not operation-level defaults)
- TypeSpec server defaults (`param?: int32 = 100`) do NOT get `??` because TCGC doesn't populate `clientDefaultValue` from TypeSpec defaults — only from the `@clientDefaultValue` decorator
- Type mismatch protection: string default on int32 param is rejected (e.g., `@clientDefaultValue("mismatch")` on `int32` doesn't emit `??`)

### Files changed
- `src/components/send-operation.tsx` — Added `applyClientDefault()`, `formatDefaultValue()`, `isDefaultValueTypeMatch()`, and supporting helpers. Modified `getParameterAccessor()`, `getHeaderAccessor()`, `getBodyAccessor()`, `getSpreadPropertyAccessor()` to apply defaults
