  numeric values bare, matching TypeScript enum syntax.

### Architecture decisions
- **Alloy ObjectExpression/ObjectProperty**: Used Alloy's structural components for the
  return object literal instead of raw `code` templates. This ensures proper indentation,
  property name quoting (via `PropertyName`), and consistent formatting with Alloy's
  rendering engine.
- **Same SourceFile**: Serializers and deserializers are rendered in the same `models.ts`
  file as type declarations. This prevents the well-known Alloy self-import bug where
  splitting declarations and their serializers causes circular imports.
- **Usage flag filtering**: Models are filtered by `UsageFlags.Input` for serializers
  and `UsageFlags.Output | UsageFlags.Exception` for deserializers, matching the legacy
  emitter's behavior.
- **Null-check pattern**: Optional/nullable properties with complex types (models, dates,
  bytes) get wrapped with `!value ? value : serialize(value)`. Simple passthrough types
  (string, number, boolean) skip the guard since undefined passes through safely.
- **Exported utilities**: `getSerializationExpression()`, `getDeserializationExpression()`,
  and `needsTransformation()` are exported for reuse by future components (e.g.,
  polymorphic serializers, operation request builders).

### Key learnings (see knowledge.md)
- Alloy's ObjectExpression with ObjectProperty handles proper indentation and property
  name quoting automatically — prefer this over raw `code` templates for object literals.
- UsageFlags is a bitmask — use `(model.usage & UsageFlags.Input) !== 0` not `===`.

### Architecture decisions
- **Switch-based routing**: Polymorphic serializers use a `switch` statement on the
  discriminator property, with each case routing to the subtype's serializer/deserializer
  via refkey. This matches the legacy emitter's pattern.
- **Discriminator access**: Serializers use client property name (`item["kind"]`) since the
  input is a typed SDK object. Deserializers use wire property name (`item["shape_type"]`)
  since the input is raw JSON from the service.
- **Replaces regular serializer**: For discriminated parent models, the polymorphic
  serializer gets `serializerRefkey(model)`, so external code that references the
  serializer automatically gets the switch-based version. Regular property serializers
  are NOT generated for discriminated parents.
- **Default case**: Returns `item` as-is for unknown discriminator values, providing
  forward compatibility when the service adds new subtypes.
- **Shared utility**: `buildPolymorphicSwitchBody()` is exported and shared between
  serializer and deserializer, parameterized by the refkey function (serializerRefkey
  vs deserializerRefkey).
- **All subtypes in switch**: The switch includes ALL entries from `discriminatedSubtypes`
  (including transitive subtypes in multi-level hierarchies), not just direct subtypes.

### Architecture decisions
- **Single-line return statement**: The return statement with `context.path(path).verb({...})`
  is rendered as a single line to avoid Alloy indentation issues with multi-line code templates.
  Alloy's raw `\n` characters in code templates don't receive context indentation, so multi-line
  formats caused inconsistent spacing.
- **expandUrlTemplate from runtime**: Added `expandUrlTemplate` to `httpRuntimeLib` instead of
  creating a static helper. The legacy emitter uses a static helper file, but for the new emitter
  treating it as a runtime import is cleaner. Task 2.5 can adjust this if needed.
- **Accept header always present**: The accept header is derived from response content types
  and is always included when the operation has typed responses. This matches the legacy emitter.
- **Parameter mapping via correspondingMethodParams**: HTTP parameters (path, query, header)
  are mapped back to method parameters via `correspondingMethodParams[0]` to determine the
  correct accessor expression (direct arg vs options bag).
- **Body serialization via refkey**: Body serialization uses `getSerializationExpression()` which
  returns Alloy Children containing refkeys. This is embedded in a `code` template so Alloy
  auto-resolves the serializer function reference and generates imports.

### Key learnings (see knowledge.md)
- Alloy `code` templates with raw `\n` produce inconsistent indentation — avoid them.
- Alloy combines imports from the same package into a single statement with inline `type`.
- Alloy `FunctionDeclaration` breaks parameters onto separate lines with 2+ params.

### Architecture decisions
- **async prop handles Promise wrapping**: Alloy's `FunctionDeclaration` with `async` prop
  automatically wraps the `returnType` in `Promise<>`. Do NOT double-wrap with
  `returnType={code\`Promise<${type}>\`}` — just provide the raw inner type.
- **Status codes as strings**: Expected status codes are formatted as quoted strings (e.g.,
  `"200"`) because `PathUncheckedResponse.status` is a string type in the HTTP runtime.
- **LRO status code expansion**: For LRO operations (non-GET), additional polling status codes
  (200, 201, 202) are added to match the legacy emitter's behavior.
- **Reuse of deserialization helpers**: Uses `getDeserializationExpression()` and
  `needsTransformation()` from the serialization module, which handle models, arrays,
  dicts, dates, bytes, and nullable types.
- **Single return path**: The function has a single return statement after the status check,
  similar to the legacy emitter's pattern. Complex multi-return paths (like dual-format
  JSON/XML responses) will be added when XML deserialization is implemented (FR7).

### Key learnings (see knowledge.md)
- Alloy `async` FunctionDeclaration wraps returnType in Promise automatically — do not double-wrap.
