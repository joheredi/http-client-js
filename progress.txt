### Architecture decisions
- **Two declarations per enum**: Every enum produces both a type alias and a Known enum,

  Azure-specific union-as-enum patterns. Tests simulate extensibility by overriding isFixed.

  numeric values bare, matching TypeScript enum syntax.

### Architecture decisions
- **Switch-based routing**: Polymorphic serializers use a `switch` statement on the
  discriminator property, with each case routing to the subtype's serializer/deserializer
  via refkey. This matches the legacy emitter's pattern.
- **Discriminator access**: Serializers use client property name (`item["kind"]`) since the
  input is a typed SDK object. Deserializers use wire property name (`item["shape_type"]`)
  since the input is raw JSON from the service.
- **Replaces regular serializer**: For discriminated parent models, the polymorphic
  serializer gets `serializerRefkey(model)`, so external code that references the
  serializer automatically gets the switch-based version. Regular property serializers
  are NOT generated for discriminated parents.

  forward compatibility when the service adds new subtypes.
- **Shared utility**: `buildPolymorphicSwitchBody()` is exported and shared between
  serializer and deserializer, parameterized by the refkey function (serializerRefkey
  vs deserializerRefkey).
- **All subtypes in switch**: The switch includes ALL entries from `discriminatedSubtypes`
  (including transitive subtypes in multi-level hierarchies), not just direct subtypes.

  apiKey/http → KeyCredential, oauth2/openIdConnect → TokenCredential. Supports union
  types for multiple auth schemes.
- **Context members vs options members**: Context interface contains non-endpoint,
  non-credential params that need to be on the client object (e.g., apiVersion). Options
  interface contains all optional params consumers can configure at creation time.
- **getClient() call pattern**: Without credentials: `getClient(endpointUrl, options)`.
  With credentials: `getClient(endpointUrl, credential, options)`.

### Key learnings (see knowledge.md)
- TCGC only adds `apiVersion` to `clientInitialization.parameters` when operations
  explicitly use an api-version query parameter. The `@versioned()` decorator alone
  only populates `client.apiVersions` array.
- Alloy's `toRenderTo` matcher calls `dedent()` internally which expects a string —
  `expect.stringContaining()` doesn't work for single-file renders. Use `renderToString()`
  + `toContain()` instead.

### Architecture decisions
- **Child clients = operation groups**: TCGC represents operation groups as child clients
  via `client.children`. Each child client has its own `methods` and potentially its own
  `children` for nested groups. The `SdkClientAccessor` pattern is deprecated in favor
  of the `parent`/`children` hierarchy.
- **Naming convention matches legacy emitter**: Interface is `{ChildName}Operations`,
  factory is `_get{ChildName}Operations`, property is `camelCase(childName)`. The
  child client's name from TCGC is used directly (e.g., `Widgets` → `WidgetsOperations`).
- **Factory function delegates via refkeys**: Each method in the factory's return object
  references the public operation function via `publicOperationRefkey(method)`, enabling
  Alloy to auto-generate cross-file imports.
- **Context flows through factory chain**: All factory functions take the root client
  context as a parameter and pass it to both operation delegates and nested factory
  calls. This mirrors the legacy emitter's pattern where the RLC client flows through
  all operation group layers.
- **BFS traversal for file generation**: Same BFS queue pattern used in `operation-files.tsx`
  for collecting operation groups. Each child client is visited with its accumulated
  prefix path, producing the correct directory nesting.

### Root Cause Analysis
The legacy emitter adds `_` prefix to model names when `type.isGeneratedName` is true (see
`normalizeModelName()` in `submodules/autorest.typescript/packages/typespec-ts/src/modular/emitModels.ts`
line 772-774). This is because multipart body types like `{ name: HttpPart<string>; file: HttpPart<bytes> }`
are anonymous inline models — TCGC generates a name for them (e.g., `UploadFileRequest`) and sets
`isGeneratedName: true`. The new emitter was missing this prefix logic.

### Architecture decisions
- **Centralized naming utility**: `getModelName` and `getModelFunctionName` in `src/utils/model-name.ts`
  are used by ALL model declaration and serializer components. This ensures consistency.
- **namekey with ignoreNamePolicy**: The `change-case` library's `pascalCase` and `camelCase` strip
  underscores entirely (`_UploadFileRequest` → `UploadFileRequest`). Using `ignoreNamePolicy: true`
  prevents this but requires manually handling camelCase for function names.
- **Applied to ALL serializer types**: Although only multipart models were failing in tests, the
  underscore prefix applies to ANY generated-name model, including JSON and XML serializers. Applied
  the fix everywhere for correctness.
